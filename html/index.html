CI/CD Pipeline Proof of Concept

Overview

This website describes the unseen infrastructure and automation that continously takes some lines of code and turns it into a working product. This project had two goals:
  1. Get hands-on experience with some really cool enterprise-grade technologies
  2. Be entirely cost-free
These goals challenged me to better understand the services and to seek out creative solutions. To find out more about me check out resume.sukoneck.com.

Pipeline

Workflow
Whenever the source is modified the pipeline will automatically build, publish, and deploy a containerized website:
  1. The master branch of the code repository is modified
  2. The modification triggers the pipeline
  3. The pipeline clones the code repository and builds a container image from it
  4. The pipeline pushes the container image to the image repository with the latest tag
  5. The pipeline tells the container hosting service to restart the application
  6. The hosting service pulls a fresh copy of the latest image from the image repository
  7. The hosting service serves the latest container
  8. The DNS and edge cache provider redirects users when they navigate to pipeline.sukoneck.com
Revisions
After the pipeline was up and running I optimized for some interesting things:
  • While doing website development I noticed that when I pushed commits to non-master branches the pipeline was still being triggered, but it is an easy change to filter triggers to a specific branch.
  • The first iteration of the pipeline used a Docker Hub feature called Automated Builds to build the container image and push it to the image repository. The Automated Build was similarly triggered from by modification to the code repository. The feature is free and convenient; however, the build took two to three minutes. Now, when the pipeline tool executes the build task, it only takes 20 seconds.
  • Normally, new code would be pushed to the host service but as I was watching logs from the hosting service, while troubleshooting an issue with cold starts, I noticed that a new container was pulled from the image repository during the start-up process. So rather than have the pipeline spend time pulling and pushing new code, I simply had it tell the hosting service to restart.

Infrastructure Components

Website

This website is being served from a docker container running Nginx. There are templates listed in the references section below. I modified the Dockerfile and removed the wrapper.sh content from the template based on what I learned here.

Repositories

The repository for the code to build this website image is GitHub. The repository for this container image is Docker Hub.

Hosting Service

The container is hosted in Azure App Service. One of the constraints of the free tier of Azure App Service is that you are unable to use the Always On feature. If the app goes unused for ~5 minutes, then it is turned off and has to go through a cold start when it is used again. The Always On feature keeps the application running by sending http requests to it every 5 minutes or so. For me, the cold start took about 30 seconds, which wasn't actually cool at all. So, I followed Ben Chartrand's recommendation and set up an Application Insights availability test, which was also free. The traffic from the availability test is enough to keep the app running and prevent that painful cold start.

One feature that is included in the free App Service is the ability to force HTTPS. This is common practice, but the ability to "flip a switch" to have HTTP traffic redirected is convenient.

I love hearing about how people choose between AWS, Azure, and GCP, so for those like me, I'll explain why I chose Azure. I knew I wanted to serve the website from a container and Microsoft offers a free tier for App Service. They also offer a free tier for Azure DevOps. The free tiers also seemed like they had workable constraints. AWS has an equivalent service called Fargate; however, AWS' "free tier" has a different model than Microsoft's that only lasts for a year and that felt like it would be cheating on goal #2. I also know how quickly a year really passes. GCP recently announced Cloud Run which has an always free tier similar to Microsoft's; however, Cloud Run is still in BETA and I would still need to use Azure DevOps. So, to keep costs actually at zero and to take advantage of that sweet ecosystem synergy, I chose Azure. That said, it's on my backlog to move over to Cloud Run later on.

Pipeline

There were really only two contenders for pipeline tool, Jenkins and Azure DevOps. I went with Azure DevOps because it's PaaS and also comes with Azure Boards, whereas Jenkins would require managing a host server and adding something like Jira or Trello.

DNS

This project was a great opportunity to experiment with Cloudflare, which seems like an awesome company. I started by migrating the nameservers for sukoneck.com from Namecheap to Cloudflare. This was done with a heavy heart, because I'm a huge Namecheap fan. Fortunately, I've got plenty of goofy domains with Namecheap to play with. With DNS now hosted in Cloudflare, I created a CNAME record, but quickly realized that wasn't going to work. I had forgotten that you have to add a custom domain to your Azure App Service to be able to use a CNAME record. You can see the custom domain error that I saw at cnameexample.sukoneck.com.

The bad news: the custom domain feature of Azure App Service is not free. So, I am stuck with the azurewebsites.net domain to keep everything free. Ok, fine, keep moving. With Namecheap I had set up a ton of URL redirect records for convenience sake, it was as simple and free as creating any other DNS record, so I decided that would be a enough of a minimally viable solution for now. Cloudflare approaches URL redirects much differently. I followed Dom J H's guidance by creating a dummy A record to trigger Cloudflare's DNS and then created a Page Rule to do the permanent redirection from pipeline.sukoneck.com to the azurewebsites.net URL. Page Rules offer powerful features like path-based routing, but as a result, you only get three free Page Rules. On the brightside, I was still able to take advantage of Cloudflare's cache service which was caching 72% of content even during site development.
